## Context

`cc-session-reader` today is a single-provider (Claude Code only) CLI: `internal/claudecodec` parses raw JSONL, `internal/summarizer`/`internal/analyzer` do deterministic compression, `internal/formatter` renders `read`/`context`, and everything is driven from `cmd/cc-session`. There is no cross-session-provider abstraction, no persistent evidence store, and no LLM-assisted summarization — by design, since the tool's whole value proposition is deterministic, lossless-by-reference compression.

This change adds multi-agent provider support (Claude Code, Codex, and Antigravity) and a new optional derived-artifact layer (the Local LLM handoff) on top, without weakening the existing guarantee: raw session data is immutable, and anything the deterministic filter throws away must remain reachable by evidence ID. The design has to keep the "core is deterministic, LLM is optional and additive" property intact — a user with no Local LLM endpoint configured must still get full parse/filter/search/expand/stats functionality.

Stakeholders: single maintainer (peterramsw) using this personally to resume Claude Code / Codex sessions across `typingnote` and other real projects; no team, no CI infra assumed yet.

Real usage topology: three agent apps — Claude Code, Codex, and Antigravity — are typically all open at once, each having launched its own `cc-session serve-mcp` process over stdio. Only one app is actively driving work at any given moment; the other two sit idle with a live MCP connection so that if the active one gets stuck (context exhausted, crashed, wedged), a different app/model can pick up the handoff and continue. This means "single user, single machine, but up to three concurrent OS processes touching the same `~/.session-context/` store" is a normal, expected runtime shape, not an edge case — the design has to handle it without Docker/daemons/distributed locking, just correct local file-concurrency handling in Go.

## Goals / Non-Goals

**Goals:**
- One `SessionProvider` abstraction so core logic (filtering, evidence, handoff, MCP) never branches on provider type.
- A handoff pipeline that is auditable end-to-end: every non-obvious claim in `handoff.json` carries an evidence ref that resolves back to real source bytes.
- MCP + Skills usable from Claude Code, Codex, and Antigravity without duplicated workflow logic. Antigravity is a first-class target for both MCP access and its own Skill package.
- The MCP server must tolerate up to three concurrent stdio server processes (one per app) against the same evidence store without deadlocking, corrupting files, or requiring the apps to coordinate with each other.
- Zero hard dependency on any Local LLM, Anthropic, OpenAI cloud API, or GB10 endpoint for the deterministic core of the tool (discover/inspect/parse/filter/search/expand/stats).
- Preserve 100% of existing upstream CLI behavior and tests.

**Non-Goals:**
- No vector DB / RAG / semantic search (exact/substring search only, v1).
- No database (SQLite/Postgres) — flat files under `~/.session-context/`.
- No autonomous mutation of the user's workspace, git state, AGENTS.md/CLAUDE.md, or deployments — `verify_workspace` and the Skills are strictly read-only/advisory.
- No attempt to support arbitrary future agent CLIs beyond the three named targets in this change — the `SessionProvider` interface should make additional adapters possible later, but Claude Code, Codex, and Antigravity are the current product scope.

## Decisions

**1. `SessionProvider` interface lives in `internal/session`, adapters are separate packages.**
`internal/session/provider.go` defines `SessionProvider` (`Name`, `Discover`, `Inspect`, `Parse`) and the normalized `SessionEvent`/`SessionMetadata`/`SessionRef` types. `internal/claudecodec` is refactored to implement this interface (its actual JSONL parsing logic is reused, only wrapped — classify.go/reader.go/model.go stay as the Claude-specific engine). `internal/codexcodec` is new and implements the same interface for Codex. `internal/antigravitycodec` follows the same pattern after the local Antigravity session format is verified. Core packages (`filtering`, `evidence`, `handoff`, `mcp`) depend only on `internal/session`'s interface types, never on provider-specific codec packages. *Alternative considered*: keep a single parser with provider branches — rejected per explicit requirement and because it would make each provider's format leak into shared code.

**2. Deterministic, content-derived IDs (no UUIDs, no counters).**
`event_id = "ev-" + hex(sha256(session_id|provider|sequence|source.path|source.line_start|source.byte_start|event_type))[:16]`. `evidence_id` is derived the same way from the evidence record's stable fields (session_id, event_id, kind). Because the hash input is the *position and type* of the event in the source, not its filtered/redacted content, IDs stay stable even if redaction rules or summarization heuristics change later — only re-running against a *changed raw session* changes IDs, which is correct (the requirement is "same unmodified session ⇒ same IDs across reruns", not "same IDs forever regardless of source"). *Alternative considered*: random UUIDs — rejected, explicitly disallowed (breaks reproducibility, breaks the "hallucinated evidence ID" detection story since real IDs would look no different from made-up ones).

**3. Evidence store is flat files, one directory per `(provider, session_id)`.**
`~/.session-context/<provider>/<session-id>/{manifest.json, normalized.jsonl, filtered.jsonl, evidence-index.json, handoff.json, handoff.md}`. `evidence-index.json` is a map keyed by `evidence_id` so expand is O(1) lookup; it stores `source_path/line_start/line_end/byte_start/byte_end` so expand re-reads the *original* raw session file on demand rather than duplicating raw content into the store (keeps the store small and keeps "raw session is the only source of truth for full content" true). All paths written under `storage_root` are validated (via `filepath.EvalSymlinks` + prefix check) to stay inside `storage_root`; all raw reads for expand are validated to stay inside the configured session roots (`allowed` roots), rejecting `..`-escapes and symlink escapes before opening any file.

**4. Redaction runs at two boundaries, never mutates stored raw bytes.**
(a) When `filtered.jsonl` is written (this is what the local LLM sees and what `get_handoff`/`search_session` return) — redaction is baked in at write time. (b) When `expand_evidence`/`cc-session expand` re-reads raw bytes on demand — redaction runs on the read path before returning content, default on, with an explicit `--unredacted`/`redacted:false` override gated by config. `normalized.jsonl` stores enough of the event (including original content) to regenerate filtered/redacted views deterministically, but is itself treated as sensitive internal state (not sent anywhere, not displayed unredacted by default). Detection is deny-list regex (API keys, bearer/access/refresh tokens, passwords, cookies, client secrets, `.env`-style assignments, GitHub tokens, Anthropic/OpenAI-shaped tokens, JWTs, PEM blocks) plus a conservative high-entropy-string heuristic for generic secrets in key=value contexts. This is necessarily best-effort; documented as such in `docs/security.md`.

**5. Handoff generation: schema validate → semantic validate → one repair attempt → fail loud.**
`internal/handoff/schema.go` defines the Go structs + JSON schema; `validator.go` runs structural validation first, then semantic rules (evidence-ref existence, unevidenced-claim demotion, deployment/rollback conflict, branch/commit mismatch). If local LLM raw output fails JSON parsing, one repair round-trip is attempted (re-prompt with the parse error); if that also fails, the raw local LLM output is saved next to `handoff.json` (e.g. `handoff.raw-failed.json`) and the command/tool returns an error — never a silently-empty handoff.

**6. Context budgeting: heuristic phase segmentation + independent chunk distillation + evidence-preserving merge.**
`internal/distiller/chunker.go` splits the filtered transcript into ordered chunks along heuristic phase boundaries (requirement/planning/design/implementation/debugging/testing/deployment/rollback/final-report), never reordering events. Each chunk is distilled independently into a partial structured state (same schema, partial). `merger.go` concatenates/dedupes list fields and unions evidence refs; it validates that every evidence ref in a partial result exists in the evidence index *before* merging it in — refs that don't resolve are dropped with a warning rather than propagated, closing the "model invents evidence during merge" loophole at the merge boundary too, not just at final validation.

**7. MCP SDK: prefer the official `modelcontextprotocol/go-sdk`, fall back to `mark3labs/mcp-go` if it proves unworkable for stdio.**
Decision finalized during Phase 13 implementation (needs a hands-on build+smoke-test, not a documentation-only choice) — tracked as an open question below, not blocking earlier phases since `internal/mcp` is written against a small internal `Tool` abstraction that either SDK can sit behind.

**8. One binary, `serve-mcp` subcommand; MCP and CLI share `internal/*` core directly (no RPC between them).**
`cmd/cc-session` gains `serve-mcp`; `internal/mcp/server.go` wires the same `internal/handoff`, `internal/evidence`, `internal/session`, `internal/workspace` packages that the CLI subcommands call directly. This satisfies "CLI 與 MCP 必須共用同一套 core" without introducing a second process/IPC boundary for what is fundamentally one machine's local tool.

**9. Skills: shared Markdown in `skills/common/`, thin per-platform entry points for all three apps.**
`skills/claude-code/SKILL.md`, `skills/codex/SKILL.md`, and `skills/antigravity/SKILL.md` are short files whose body is "read `skills/common/{resume,close,review}-session.md` and follow it" plus whatever platform-specific frontmatter/invocation convention each tool requires:
- Claude Code: `allowed-tools` frontmatter, installed to `~/.claude/skills/cc-session-handoff/SKILL.md` (as the existing root `SKILL.md` already demonstrates for the base tool).
- Codex: whatever its skill/prompt convention is — confirmed during Phase 14, not assumed here.
- Antigravity 2.0: also uses the filename `SKILL.md` (confusingly, same name as Claude Code's convention) but a different install location (`~/.gemini/skills/<skill-name>/SKILL.md`, global) and different frontmatter — notably a `tools` field listing which MCP server tools (e.g. `mcp_session-context_*`) the skill may call, confirmed via Google's published Agent Skills documentation rather than assumed.

Content lives once in `skills/common/`; drift is structurally impossible since there's only one copy of the workflow text. Because Claude Code and Antigravity both expect a file literally named `SKILL.md`, each platform's version must live in its own subdirectory (`skills/claude-code/SKILL.md` vs `skills/antigravity/SKILL.md`) — they can never be the same physical file even though the filename matches, since their frontmatter schemas differ.

**10. Concurrency safety: atomic writes for every artifact file, short-lived advisory locks only around read-modify-write sections, never around slow operations.**
Every write to a JSON artifact (`manifest.json`, `evidence-index.json`, `handoff.json`) is done by writing to a temp file in the same directory and `os.Rename`-ing over the target — atomic on both Windows (same-volume rename) and Linux, so a concurrent reader from another process never observes a half-written file. Read-modify-write sequences that aren't naturally atomic (e.g. "create handoff if one doesn't already exist for this session, or reuse it unless `--force`") are protected by a per-session advisory lock file (`<session-dir>/.lock`, acquired via `flock`-equivalent, e.g. `github.com/gofrs/flock`, which works cross-platform) held only for the duration of the local check-and-write, *not* across the local LLM HTTP call itself — the distiller call happens before the lock is (re-)acquired for the final write, so a slow/hung LLM call in one process never blocks another process's unrelated reads or even its own handoff creation for a *different* session. Lock acquisition uses a short timeout (a few seconds) with a small number of retries; if it still can't acquire the lock (another process is mid-write to the *same* session), the caller gets a clear "handoff generation already in progress for this session, try again shortly" error rather than hanging indefinitely — there is no scenario in the intended usage pattern (only one app actively working at a time) where contention should be sustained. Pure reads (`list_sessions`, `get_handoff`, `search_session`, `expand_evidence`) never take locks; they only ever read fully-written files thanks to the atomic-rename write pattern. *Alternative considered*: a single shared daemon process that all three apps connect to (removing multi-process contention entirely) — rejected as unnecessary complexity (extra lifecycle management, port/socket allocation, discovery) for a problem that plain file-level locking already solves, and it would violate the "no daemon/Docker" constraint's spirit.

## Risks / Trade-offs

- [Risk] Real Codex session file format is unknown until investigated → [Mitigation] Phase 3 is dedicated real-machine investigation before writing `codexcodec`; if this machine has no Codex sessions, a fixture is built and explicitly labeled synthetic in test names/docs, and `docs/session-provider.md` records exactly which schema/version was targeted.
- [Risk] Go MCP SDK ecosystem is young; the chosen SDK may lack features or churn its API → [Mitigation] `internal/mcp` isolates SDK usage behind a thin internal interface; smoke-tested before commit; fallback candidate identified up front.
- [Risk] Regex/entropy-based redaction misses novel secret shapes → [Mitigation] default-deny-by-pattern is inherently incomplete; document known gaps in `docs/security.md`, bias toward over-redaction, and never treat redaction as a substitute for not putting secrets in session logs in the first place.
- [Risk] Deterministic IDs could collide across very large corpora → [Mitigation] 16 hex chars (64 bits) of SHA-256 over a fairly specific input tuple is collision-safe for realistic session/evidence counts; not cryptographically hardened against adversarial input, which is fine for a local single-user tool.
- [Risk] Local LLM chunking/merge still produces plausible-but-wrong claims even with evidence refs pointing to real-but-misread evidence → [Mitigation] this is inherent to any LLM-in-the-loop design; addressed at the workflow level (Resume Skill mandates `verify_workspace` + re-verification of anything the current task depends on), not solvable purely in the validator.
- [Risk] Windows-first dev machine, Linux support only exercised by code review not real Linux Codex/Claude sessions → [Mitigation] path/config resolution isolated in one module, unit-tested against both path styles even without a real Linux session available.
- [Risk] Three concurrent `serve-mcp` processes (Claude Code + Codex + Antigravity) racing on the same session's evidence directory → [Mitigation] atomic rename-based writes for all artifacts + short-lived per-session advisory lock only around the create-handoff critical section, tested explicitly with a multi-process concurrency test (parallel `go test` goroutines or subprocesses hammering `create_handoff`/`filter_session` for the same session); no lock is ever held across the local LLM HTTP call, so one app's slow/stuck LLM call cannot block another app's unrelated work.

## Migration Plan

Purely additive: no existing data migrates, no existing command's behavior changes. Rollout is "merge behind new subcommands/config keys that don't exist today." Rollback is a plain `git revert` — the evidence store under `~/.session-context/` is new state owned entirely by this feature and safe to delete if the feature is abandoned. The deterministic provider/filter/search/expand path is always available; Local LLM handoff generation is disabled unless `local_llm.enabled` and `local_llm.base_url` are configured or explicitly overridden.
